# S3 Bucket Provisioning

## Purpose

MemoriaHub stores all media files in the `marin-memoriahub` S3 bucket (region `us-east-1`). The API uploads objects via presigned multipart URLs and serves them via presigned download URLs. The bucket is **private** — public access is fully blocked; every read or write goes through short-lived signed URLs generated by the API.

## What `setup-bucket.cjs` configures

The script is idempotent and safe to re-run at any time. On each run it applies:

| Step | What it does |
|------|-------------|
| Create bucket | Creates `S3_BUCKET` in `S3_REGION` if it does not already exist. |
| Block public access | Sets all four `PublicAccessBlock` flags to `true` — no ACL or policy can ever make objects public. |
| Default encryption | Enables SSE-S3 (AES256) as the bucket default; every object is encrypted at rest automatically. |
| CORS | Applies the rules from `cors.json` (merged with `APP_URL` and `CORS_EXTRA_ORIGINS`) so browsers can `PUT` multipart parts directly to S3 and read the `ETag` response header required to complete the upload. |
| Lifecycle rule | Aborts incomplete multipart uploads after 7 days, preventing orphaned parts from accumulating storage charges. |

## How to run

Run from the repo root after sourcing the environment file:

```bash
set -a; source infra/compose/.env; set +a
NODE_PATH=apps/api/node_modules node infra/aws/setup-bucket.cjs
```

The script resolves `@aws-sdk/client-s3` from `apps/api/node_modules` — no separate install needed.

### Environment variables read by the script

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `S3_BUCKET` | yes | — | Bucket name (e.g. `marin-memoriahub`) |
| `S3_REGION` | no | `us-east-1` | AWS region |
| `AWS_ACCESS_KEY_ID` | yes | — | AWS credentials |
| `AWS_SECRET_ACCESS_KEY` | yes | — | AWS credentials |
| `S3_ENDPOINT` | no | — | Override endpoint for MinIO / LocalStack |
| `APP_URL` | no | — | Merged into CORS `AllowedOrigins` |
| `CORS_EXTRA_ORIGINS` | no | — | Comma-separated additional origins merged into CORS |

## CORS origins

`cors.json` is the source of truth for allowed origins. The defaults are:

| Origin | Purpose |
|--------|---------|
| `https://memoriahub.dev.marin.cr` | Production deployment on the dev VPS |
| `http://localhost:3535` | Local development via Nginx |
| `http://localhost:5173` | Local Vite dev server (direct) |

To add more origins, either edit `cors.json` directly or set `CORS_EXTRA_ORIGINS` as a comma-separated list before running the script. The script deduplicates entries, so re-running with an origin already in `cors.json` is harmless.

### Why `ExposeHeaders: ETag` is required

When the browser uploads individual parts of a multipart upload directly to S3 via presigned `PUT` URLs, S3 returns an `ETag` in the response header for each part. The API must collect these per-part ETags and pass them back to S3 in the `CompleteMultipartUpload` call. Without `ExposeHeaders: ["ETag"]` in the CORS configuration, the browser's `fetch`/XHR machinery strips the `ETag` header from the response before JavaScript can read it, which makes it impossible to complete the upload.

The full set of exposed headers configured in `cors.json`:

```json
"ExposeHeaders": ["ETag", "x-amz-request-id", "x-amz-id-2"]
```

## Verifying the configuration

After running the script, confirm CORS is active with a preflight check against the live bucket.

**Allowed origin — should return `Access-Control-Allow-Origin` and `Access-Control-Expose-Headers: ETag`:**

```bash
curl -s -D - -o /dev/null -X OPTIONS \
  "https://marin-memoriahub.s3.amazonaws.com/uploads/x.jpg" \
  -H "Origin: https://memoriahub.dev.marin.cr" \
  -H "Access-Control-Request-Method: PUT"
```

Expected response headers include:

```
Access-Control-Allow-Origin: https://memoriahub.dev.marin.cr
Access-Control-Allow-Methods: GET, PUT, POST, HEAD
Access-Control-Expose-Headers: ETag, x-amz-request-id, x-amz-id-2
```

**Disallowed origin — should return `403`:**

```bash
curl -s -o /dev/null -w "%{http_code}" -X OPTIONS \
  "https://marin-memoriahub.s3.amazonaws.com/uploads/x.jpg" \
  -H "Origin: https://evil.example.com" \
  -H "Access-Control-Request-Method: PUT"
```

Expected: `403`

## Security notes

- **Private bucket**: all four `PublicAccessBlock` flags are enabled; no object can be made public through ACLs or bucket policies.
- **Encryption at rest**: SSE-S3 (AES256) is applied to every object by default; no application-level changes are required.
- **Credentials via environment only**: `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` are sourced from `infra/compose/.env`, which is git-ignored. Credentials are shared from the knecta stack on the dev VPS.
- **Dedicated bucket**: `marin-memoriahub` is used exclusively by MemoriaHub; no other project shares it.
- **Short-lived signed URLs**: the API issues presigned upload and download URLs via `@aws-sdk/s3-request-presigner`. The default expiry is 1 hour (`SIGNED_URL_EXPIRY=3600`); no credentials are ever exposed to the browser.
